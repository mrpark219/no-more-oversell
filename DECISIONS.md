# DECISIONS.md

이 문서는 선착순 숙소 예약/결제 서버의 주요 설계 결정을 정리한다.

## 프로젝트 목표

- 특정 시각에 오픈되는 제한 수량 숙소 상품을 미달 및 초과 판매 없이 판매한다.
- 모든 사용자에게 동등한 기회를 제공한다.
- 2대 이상의 애플리케이션 서버가 동시에 요청을 받아도 동일한 정합성 기준으로 처리한다.
- 결제 실패, 재고 소진, 중복 요청, 처리 중 주문, 잘못된 결제 조합을 구분해 응답한다.
- 결제수단 추가 시 주문 생성 흐름의 변경을 최소화한다.

## 사용 기술

- Java 21
- Spring Boot 3.5
- MySQL 8.4
- Redis 8.6.3
- Spring Data JPA
- QueryDSL
- Flyway
- Resilience4j Circuit Breaker
- Testcontainers
- K6

# 1. API 흐름

### 결정

- 예약 흐름은 `GET /api/checkout`과 `POST /api/orders` 두 API로 구성한다.
- `GET /api/checkout`
    - `userId` header와 `stayProductId` query parameter를 받는다.
    - 상품 상태와 재고를 조회한다.
    - 재고가 없으면 주문서 토큰을 생성하지 않고 `SOLD_OUT`으로 응답한다.
    - 재고가 있으면 사용자 포인트를 조회하고 주문서 토큰을 생성해 반환한다.
- `POST /api/orders`
    - 주문서 토큰, 상품 ID, 결제수단별 금액을 입력받는다.
    - 주문서와 상품 상태를 검증한다.
    - 결제 조합과 총 결제 금액을 검증한다.
    - 결제를 승인한다.
    - 재고 차감, 구매 제한 검증, 결제 저장, 주문 생성을 수행한다.
- 현재 구현은 실제 PG redirect나 callback 없이 mock gateway 기반 동기 승인 방식으로 동작한다.

### 고려한 대안

- 결제 준비 API와 PG callback API를 분리한다.
    - 장점
        - 실제 PG 연동 구조에 가깝다.
        - callback 중복, 지연, 만료 처리를 명확히 다룰 수 있다.
    - 제외한 이유
        - callback 흐름까지 구현하면 핵심인 재고 정합성 처리보다 범위가 커진다.
- checkout을 `POST`로 만든다.
    - 장점
        - 주문서 토큰 생성까지 포함하므로 HTTP 의미상 더 엄격하다.
    - 제외한 이유
        - 현재 checkout은 조회 성격이 더 강하다.
        - 상품 정보, 재고, 포인트를 함께 조회하는 흐름이 자연스럽다.
- checkout에서는 주문서를 만들지 않고 Booking API에서만 생성한다.
    - 장점
        - checkout을 순수 조회 API로 유지할 수 있다.
    - 제외한 이유
        - 주문 요청 기준점이 약해진다.
        - checkout 단계에서 주문서 토큰을 발급하는 편이 멱등성 처리에 유리하다고 판단했다.

### 한계

- `GET /api/checkout`은 주문서 토큰을 생성하므로 완전한 순수 조회 API는 아니다.
- 운영 환경에서는 결제 준비, callback 확정, 결제 만료, 중간 상태 회수 등을 별도 흐름으로 분리할 필요가 있다.

# 2. 재고 정합성

### 결정

- 최종 재고 원천은 MySQL `inventory` 테이블로 관리한다.
- DB unique/check 제약으로 재고 정합성을 보장한다.
- 재고 차감은 JPA pessimistic write lock 기반으로 처리한다.
- Redis는 품절 여부 확인용 캐시로만 사용한다.
- 재고 차감은 결제 승인 이후 주문 확정 트랜잭션에서 수행한다.

### 고려한 대안

- Redis atomic counter를 사용한다.
    - 장점
        - 빠른 재고 차감이 가능하다.
        - DB lock 경합을 줄일 수 있다.
    - 제외한 이유
        - Redis와 DB 간 수량 불일치 복구가 필요하다.
        - Redis 장애 시 최종 재고 기준이 복잡해진다.
        - MySQL 단일 재고 원천 구조가 더 단순하다고 판단했다.
- Redis Lua 기반 선점 후 DB 비동기 반영을 사용한다.
    - 장점
        - 높은 TPS에서 DB 병목을 줄일 수 있다.
    - 제외한 이유
        - 메시지 중복, 실패 복구, 보상 정책까지 함께 설계해야 한다.
        - 현재 범위에서는 구조 복잡도가 크다고 판단했다.
- Redisson 분산락을 사용한다.
    - 장점
        - 여러 서버에서 동일한 락 기준으로 처리할 수 있다.
    - 제외한 이유
        - TTL, 락 유실, Redis 장애 대응을 추가로 설계해야 한다.
        - 최종 DB 제약은 여전히 필요하다.
- 애플리케이션 `synchronized`를 사용한다.
    - 장점
        - 단일 서버에서는 단순하다.
    - 제외한 이유
        - 다중 서버 환경에서는 정합성을 보장할 수 없다.

### 한계

- 단일 상품 재고 row가 병목이 될 수 있다.
- 운영 환경에서는 DB lock wait, deadlock, connection pool 사용률 모니터링이 필요하다.
- 더 큰 트래픽에서는 Redis 기반 선점, 비동기 queue, read/write 분리 등을 추가 검토해야 한다.

# 3. 주문서 상태와 멱등성

### 결정

- 주문서 상태를 멱등성 기준으로 사용한다.
- 주문서 상태는 `CREATED`, `APPROVING`, `CONFIRMED`, `FAILED`, `EXPIRED`로 관리한다.
- Booking API는 주문서 row를 비관적 락으로 조회하고 상태를 검증한다.
    - `CREATED`이면 결제를 진행한다.
    - `APPROVING`이면 진행 중 응답을 반환한다.
    - `CONFIRMED`이면 기존 주문 결과를 반환한다.
    - 그 외 상태는 요청을 거부한다.
- DB unique 제약으로 주문과 결제 중복 생성을 방지한다.
    - `orders.order_sheet_id`
    - `payment.order_sheet_id`
- 애플리케이션 검증이 실패해도 DB 제약으로 최종 중복 생성을 막는다.

### 고려한 대안

- 클라이언트 idempotency key를 별도로 받는다.
    - 장점
        - 클라이언트가 중복 요청 기준을 직접 관리할 수 있다.
    - 제외한 이유
        - checkout 단계에서 이미 `orderSheetToken`을 발급한다.
        - 주문서 토큰과 주문서 상태를 기준으로 처리하는 편이 단순하다고 판단했다.
- Redis TTL key로 중복 요청을 막는다.
    - 장점
        - 짧은 시간의 중복 요청을 빠르게 차단할 수 있다.
    - 제외한 이유
        - Redis 장애 시 멱등성 기준이 흔들릴 수 있다.
        - 주문서 실제 상태와 Redis key 상태가 어긋날 수 있다.
        - DB 상태와 unique 제약 기반 처리가 더 안전하다고 판단했다.
- 주문 확정을 완전 비동기로 처리한다.
    - 장점
        - queue 기반으로 중복 요청을 흡수할 수 있다.
    - 제외한 이유
        - 중간 상태 관리와 복구 흐름이 추가로 필요하다.
        - 현재 범위에서는 동기 API 구조가 더 단순하다고 판단했다.

### 한계

- 오래된 `APPROVING` 상태를 자동 회수하지 않는다.
- 결제 승인 이후 서버 장애 상황을 완전히 복구하지 못한다.
    - 따라서 결제 서버와의 대사 과정 및 복구 처리가 필요하다.
- `EXPIRED` 상태의 실제 만료 전이 로직은 구현하지 않았다.

# 4. 결제 흐름과 확장성

### 결정

- 결제수단은 `PaymentMethodHandler`로 분리한다.
- 현재 결제수단은 `CARD`, `Y_PAY`, `POINT`를 지원한다.
- 주문 서비스는 결제수단별 승인/취소 구현을 직접 알지 않는다.
- `PaymentMethodHandlerFinder`가 결제수단에 맞는 handler를 찾는다.
- 결제 조합 검증은 `PaymentPlanValidator`와 `PaymentCompositionPolicy`에서 처리한다.
    - 총 결제 금액은 판매가와 같아야 한다.
    - 결제 금액은 0보다 커야 한다.
    - 같은 결제수단은 중복 사용할 수 없다.
    - PG 결제수단은 동시에 하나만 사용할 수 있다.
    - `CARD + POINT`, `Y_PAY + POINT`, `CARD`, `Y_PAY`, `POINT` 조합은 허용한다.
    - `CARD + Y_PAY` 조합은 허용하지 않는다.

### 고려한 대안

- 주문 서비스 내부에서 결제수단을 `if` 또는 `switch`로 분기한다.
    - 장점
        - 초기 구현이 단순하다.
    - 제외한 이유
        - 결제수단 추가 시 주문 서비스 변경 범위가 커진다.
        - 승인/취소 로직이 주문 서비스에 누적된다.

### 한계

- 결제수단이 늘어나면 `PaymentMethodHandler` 역할이 커질 수 있다.
- 실제 운영 환경에서는 PG 준비, 승인, 취소, 정산 등을 역할별로 더 세분화할 필요가 있다.

# 5. 결제 실패와 보상

### 결정

- 실패 지점에 따라 보상 방식을 다르게 처리한다.
- 결제 승인 실패 시 이미 승인된 결제를 역순으로 취소한다.
- 결제 승인 후 재고 부족이 발생하면 승인된 결제를 취소하고 주문 생성을 중단한다.
- 결제 승인 후 구매 제한 초과가 발생하면 승인된 결제를 취소하고 주문 생성을 중단한다.
- 결제 취소 실패 시 결제 상태를 `CANCEL_FAILED`로 저장한다.
- 주문서 상태와 실패 사유를 함께 저장해 후속 대응이 가능하도록 한다.

### 고려한 대안

- 모든 실패를 동일한 rollback 정책으로 처리한다.
    - 장점
        - 구현 흐름이 단순해진다.
    - 제외한 이유
        - 결제 승인 전과 승인 후는 외부 결제 상태가 다르다.
        - 승인 후 실패에는 명시적인 취소가 필요하다.
- 재고를 먼저 차감하고 결제 실패 시 복구한다.
    - 장점
        - 결제 성공 후 품절 가능성을 줄일 수 있다.
    - 제외한 이유
        - 재고 복구 실패 시 수량 불일치가 발생할 수 있다.
        - 재고 차감과 복구가 반복되면 복잡도가 커진다.
        - 주문 확정 시점에만 재고를 차감하는 편이 단순하다고 판단했다.
- 모든 보상을 비동기로 처리한다.
    - 장점
        - 사용자 응답 시간을 줄일 수 있다.
    - 제외한 이유
        - 승인 이후 실패는 즉시 취소가 필요한 상황이다.
        - 현재는 요청 흐름 안에서 즉시 보상 처리하는 구조를 선택했다.

### 한계

- 현재 보상은 요청 흐름 안에서 인라인으로 수행된다.
    - 결제 승인 직후 서버 장애가 발생하면 보상이 실행되지 못할 수 있다.
    - 보상 처리까지 동기적으로 수행하므로 응답 시간이 길어질 수 있다.
    - 운영 환경에서는 이벤트 또는 메시지 기반의 비동기 취소 구조를 고려해야 한다.
- `CANCEL_FAILED` 상태를 재시도하는 worker는 구현하지 않았다.
- 운영 환경에서는 결제 대사, 취소 재시도, 실패 알림, 운영 처리 절차가 추가로 필요하다.

# 6. Redis 사용 범위와 장애 대응

### 결정

- Redis는 핵심 재고 정합성 원천으로 사용하지 않는다.
- Redis는 checkout 응답, 상품 조회, 포인트 조회, 품절 여부 캐시에 사용한다.
- Redis 접근은 Resilience4j Circuit Breaker로 보호한다.
    - Redis 장애가 길어질 때 매 요청이 Redis timeout을 기다리면 checkout 응답 지연이 누적되고 DB fallback도 늦어진다.
    - Circuit Breaker를 두면 Redis 실패가 반복될 때 빠르게 fallback으로 전환해 캐시 장애가 예약 흐름 전체 장애로 번지는 것을 줄일 수 있다.
    - 단순 `try-catch`만으로도 예외 처리는 가능하지만, 연속 실패 감지와 open/half-open 상태 관리를 직접 구현해야 하므로 검증된 라이브러리를 사용하는 편이 비용 대비 효과가 크다고 판단했다.
- Redis 조회 실패 시 DB 조회로 fallback한다.
- 품절 캐시 조회 실패 시에는 품절로 단정하지 않고 DB 검증 기회를 남긴다.
- Redis 쓰기는 비동기로 처리한다.
- 캐시 쓰기 큐가 가득 차면 캐시 저장만 생략하고 요청은 계속 처리한다.

### 고려한 대안

- Redis를 재고 원천으로 사용한다.
    - 장점
        - 빠른 재고 차감이 가능하다.
    - 제외한 이유
        - Redis와 DB 간 정합성 복구가 필요하다.
        - 현재는 MySQL 단일 재고 원천 구조가 더 단순하다고 판단했다.
- Redis 장애 시 checkout API를 실패 처리한다.
    - 장점
        - 장애 상황을 빠르게 드러낼 수 있다.
    - 제외한 이유
        - Redis는 성능 보조 계층이다.
        - Redis 장애만으로 예약 흐름 전체를 중단하지 않도록 DB fallback 구조를 선택했다.
- Redis를 고가용성 구조로 구성한다.
    - 장점
        - Redis 장애 가능성을 줄일 수 있다.
        - 캐시 가용성과 응답 안정성을 높일 수 있다.
    - 제외한 이유
        - 현재 범위에서는 Redis를 핵심 정합성 계층으로 사용하지 않는다.
        - 단일 Redis와 fallback 구조만으로도 요구사항 검증이 가능하다고 판단했다.

### 한계

- Redis 장애 시 DB 부하가 증가할 수 있다.
- 운영 환경에서는 cache evict, circuit breaker, executor reject 모니터링이 추가로 필요하다.

# 7. 공정성

### 결정

- 엄격한 FIFO 큐는 구현하지 않는다.
- 모든 서버가 동일한 MySQL 재고 row lock 기준으로 경쟁한다.
- 서버별 로컬 락은 사용하지 않는다.
- 최종 재고 차감은 DB row lock 기반으로 처리한다.
- 인당 구매 제한은 주문 확정 트랜잭션 안에서 검증한다.

### 고려한 대안

- Kafka 기반 FIFO 큐를 사용한다.
    - 장점
        - 요청 순서를 명확히 관리할 수 있다.
        - 트래픽을 queue에 흡수할 수 있다.
    - 제외한 이유
        - 중복 소비, 재처리, 만료 정책 등을 함께 설계해야 한다.
        - 현재는 재고 정합성과 주문 확정을 우선했다.
- Redis queue 기반 대기열을 사용한다.
    - 장점
        - 비교적 가볍게 순번 처리가 가능하다.
    - 제외한 이유
        - Redis 장애 시 순서와 상태 복구가 필요하다.
        - 최종 재고 검증은 여전히 DB에서 수행해야 한다.
- checkout 시점에 재고를 선점한다.
    - 장점
        - 결제 성공 후 품절 가능성을 줄일 수 있다.
    - 제외한 이유
        - 미결제 사용자로 인해 재고가 장시간 점유될 수 있다.
        - 선점 만료와 회수 정책이 추가로 필요하다.
        - OTA 특성상 재고의 주체가 자기 자신이 아닌 경우가 있어 미리 확보가 어렵다

### 한계

- 실제 처리 순서는 네트워크 지연과 DB lock 획득 순서에 영향을 받을 수 있다.
- 완전한 FIFO를 보장하지는 않는다.
- 인당 구매 제한은 독점 구매를 줄이는 장치이지 FIFO 보장은 아니다.
- 엄격한 공정성이 필요하면 queue, 순번 만료, 중복 진입 방지 구조가 추가로 필요하다.

# 8. 고가용성과 TPS 급증 대응

### 결정

- checkout은 Redis 캐시를 활용해 반복 조회 비용을 줄인다.
- Redis 장애 시 DB 조회로 fallback한다.
- 결제 승인 호출은 재고 DB lock 안에서 수행하지 않는다.
- 재고 차감과 주문 생성 트랜잭션은 짧게 유지한다.
- DB unique/check 제약을 최종 방어선으로 사용한다.
- K6 부하 테스트로 checkout과 order 흐름을 검증한다.

### 고려한 대안

- 모든 요청을 queue에 넣고 단일 consumer가 순차 처리한다.
    - 장점
        - 정합성과 처리 순서를 설명하기 쉽다.
    - 제외한 이유
        - 단일 consumer가 병목이 될 수 있다.
        - consumer 장애 시 전체 주문 확정이 지연될 수 있다.
        - 현재는 여러 서버가 동일한 DB lock 기준으로 처리하는 구조를 선택했다.
- Redis 기반 선점 계층을 둔다.
    - 장점
        - DB lock 병목을 줄일 수 있다.
    - 제외한 이유
        - Redis 장애 대응과 DB 정합성 복구 정책이 추가로 필요하다.
        - 현재는 MySQL 정합성을 우선했다.

### 한계

- 현재 구조는 단일 MySQL 재고 row lock에 정합성이 집중된다.
- 더 큰 트래픽에서는 connection pool, lock wait, read/write 분리, queue 기반 구조 등을 추가 검토해야 한다.

# 9. 운영 서비스로 가기 위해 고려해야 하는 점

### 현재 구현 범위

- 현재 구현은 다음 핵심 흐름을 검증하는 데 집중한다.
    - 초과 판매 방지
    - 결제 조합 검증
    - 중복 주문 방지
    - Redis fallback
    - 결제 실패 보상
    - 부하 테스트를 통한 최종 DB 상태 검증

### 이번 버전에서 해보고 싶었지만 제외한 것

- 실제 PG 흐름
    - PG callback API
    - 결제 준비 및 redirect URL 반환
    - callback 중복 수신 처리
    - callback 지연 처리
- 주문서 생명주기 관리
    - 주문서 만료 정책
    - 오래된 `APPROVING` 주문서 회수 스케줄러
- 결제 복구 자동화
    - `CANCEL_FAILED` 재시도 worker
    - PG 결제 상태 대사
- 감사와 추적성
    - 포인트 사용/복원 이력 테이블
    - 결제 감사 로그 세분화
- 운영 관측성
    - Micrometer metric
    - 운영 dashboard
    - alert 구성

### 더 높은 트래픽에서 해보고 싶은 것

- Redis Lua 기반 재고 선점
    - DB hot row lock 병목을 줄일 수 있다.
    - 선점 만료와 DB 대사 정책을 함께 설계해야 한다.
- Kafka 또는 Redis queue 기반 대기열
    - 순간 트래픽을 완충할 수 있다.
    - 사용자 대기 UX, consumer 장애, 중복 소비 처리가 필요하다.
- 주문 확정 비동기 처리
    - API 응답 지연을 줄이고 확정 처리를 worker로 분리할 수 있다.
    - 주문 상태 조회 API와 실패 재처리 정책이 필요하다.
- PG 취소 재시도 queue
    - `CANCEL_FAILED` 상태를 자동 재시도할 수 있다.
    - 재시도 한도와 운영 알림 기준이 필요하다.
- read replica 기반 조회 부하 분리
    - checkout 조회 부하를 분산할 수 있다.
    - 최신 재고 판단은 여전히 primary DB 기준으로 유지해야 한다.
- DB connection pool과 lock wait timeout 튜닝
    - 트래픽 증가 시 가장 먼저 조정할 수 있는 현실적인 운영 파라미터다.
    - 서버 수와 DB connection 상한을 함께 봐야 한다.

### 운영 서비스 전 보강해야 하는 것

- 모니터링 시스템
    - 오래된 중간 상태가 쌓이는지 확인해야 한다.
    - 결제 취소 실패가 발생했을 때 운영자가 바로 알 수 있어야 한다.
    - 재고 수량과 확정 주문 수가 어긋나는지 지속적으로 감지해야 한다.
- 운영 작업
    - 주문 완료/실패 시 checkout cache evict
    - 오래된 `APPROVING` batch 회수 작업
    - `CANCEL_FAILED` 재시도 작업
    - PG 거래 내역 대사 작업

### 현재의 한계

- 오래된 `APPROVING` 주문서를 자동 회수하지 않는다.
- `CANCEL_FAILED` 결제를 자동 재처리하지 않는다.
- PG 승인 직후 서버 장애 상황은 현재 구조만으로 완전히 복구하지 못한다.
- 운영 환경에서는 Outbox 패턴과 이벤트 기반 복구 구조를 통해 결제 승인 이후 상태를 안정적으로 전달하고 재처리할 수 있어야 한다.
- 결제 상태 대사 작업이 없어서 외부 PG 상태와 내부 payment 상태가 어긋났을 때 자동 감지하지 못한다.
- 포인트 사용/복원 이력이 없어 감사 추적성이 부족하다.
- 트래픽이 더 높아지면 현재 DB row lock 중심 구조는 병목이 될 수 있다.
    - 이 경우 Redis 선점, queue 기반 확정, read replica, DB 튜닝을 단계적으로 검토해야 한다.
