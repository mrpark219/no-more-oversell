# AGENTS.md

## 기본 원칙

- 이 저장소는 제한된 수량의 숙소 상품을 선착순으로 예약/결제하는 서버 애플리케이션이다.
- 변경 전에는 주변 코드의 기존 패턴을 먼저 확인하고 그 스타일을 따른다.
- 저장소, 코드, 문서에는 인증 토큰, 전화번호, 이메일, 기기 ID 같은 민감정보를 남기지 않는다.
- 설계 의사결정은 `DECISIONS.md`을 참고한다.

## 코드 스타일

- 비즈니스 규칙과 상태 전이는 service 또는 domain 객체에 둔다.
- 재고, 주문, 결제 정합성은 애플리케이션 검증만 믿지 말고 DB unique/check 제약을 함께 둔다.
- Redis는 장애 가능성을 전제로 사용하고, 핵심 정합성 원천으로 사용하지 않는다.
- 실패 응답은 품절, 중복 요청, 처리 중 주문, 결제 실패, 잘못된 결제 조합, 주문서 만료처럼 원인을 구분한다.
- 핵심 로직 코드에 장애 상황 대응을 위한 로그를 추가한다.

## 테스트 작성

- 도메인 단위 테스트와 통합 테스트를 함께 작성한다.
- 테스트 메서드명은 영어로 작성한다.
- 테스트 의도는 한글 `@DisplayName`으로 명시한다.
- 테스트 코드는 given/when/then 구조로 작성한다.
- given/when/then은 주석 또는 빈 줄로 구분해 읽기 쉽게 만든다.
- 동시성 테스트는 성공 개수, 실패 개수, 최종 DB 상태를 함께 검증한다.

## 커밋 규칙

- 커밋 메시지는 `init:`, `feat:`, `fix:`, `docs:`, `test:`, `chore:` 같은 prefix를 붙인다.
- 커밋 메시지 설명은 한국어로 작성한다.
- 한 커밋에는 하나의 의도만 담는다.
